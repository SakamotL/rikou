## 一、🔴 大问题（优先处理）

### 1. 数据丢失风险 —— 产品信任的"阿喀琉斯之踵"
- **现状**：所有数据仅存 `localStorage('rikou_v2')`，无云同步、无自动备份。
- **暴露点**：
  - 换手机 / 清浏览器数据 / Safari 无痕模式 → 数据**全丢**，且 `save()` 在 `localStorage` 不可用时只 `toast` 一次，用户可能以为存了实际没存。
  - 唯一兜底是完全手动的"导出 JSON"。新用户不知道要先导出。
- **为什么是大问题**：一个叫"生活中枢"、存日记/账本/待办的产品，若数据静默丢失，用户信任直接崩塌。
- **建议（app 化必做）**：① 增加"自动本地备份到文件/IndexedDB 版本快照"；② 给出云同步方案（Supabase/Firebase，见架构章）；③ 在设置页明示"数据仅在本机，换设备请先导出"。

### 2. 账户余额是"持久化快照"而非"由流水计算" —— 财务数据不可信隐患
- **现状**：`saveLed`/`delLed` 直接 `account.balance += / -=` 并存储余额；余额是**被持久化的派生量**。
- **暴露点**：
  - 没有"按流水重算余额"的入口。一旦某笔历史账单被错误编辑、或导入了脏数据，余额会**越错越偏**且永远无法自纠。
  - 编辑一笔历史账单时回滚逻辑依赖 `old` 值正确，若 `old` 本身已错，新错叠旧错。
- **为什么是大问题**：记账软件的核心是"数要对得上"。当前架构下长期必然出现"流水总和 ≠ 账户余额"且用户无解。
- **建议**：增加 `recomputeBalances()`（遍历 ledger 重算每个账户余额），并在设置里提供"校准余额"按钮；余额改为展示时计算，或至少提供一键重算。

### 3. 提醒功能"承诺 vs 能力"错位 —— 当成闹钟会用崩
- **现状**：提醒靠 `setInterval` 每 15s 轮询 + 网页必须**保持打开**。PWA 添加到主屏幕后，若系统杀掉进程，提醒**不触发**（无 Push API / 后台任务）。
- **暴露点**：设置页只写"用于提醒弹窗"，**没告知用户"需保持网页打开/常驻"**。用户会把它当闹钟，漏事。
- **建议**：① 设置页明确提示限制；② 若要真·提醒，app 化时接入系统通知（Capacitor LocalNotifications）或 Web Push（需后端）；③ 至少加"今日待办/提醒一览"让用户在打开时一眼看到。

## 二、🟠 隐形问题（不易察觉，但会咬人）

### 4. 删除账户后，历史账单"悬挂"
- `del-acct` 删除账户，但 `ledger[].account` 仍指向旧 id。`acct(id)` 返回 `undefined`，显示时账户名为空、余额回滚时 `if(a)` 跳过——**静默断裂**，不报错但数据关联丢失。
- 建议：删除账户时弹窗让用户选择"删除关联流水 / 改挂到某账户 / 保留为'已删除账户'"。

### 5. localStorage 5MB 上限，长期用户会撞墙
- `ledger` / `notes` 无上限增长（只有 `netHistory` 截到 365 天）。多年日积月累（尤其日记/账单）极易超 5MB，`save()` 抛 quota 异常后**后续所有写入静默失败**。
- 建议：app 化时迁移 IndexedDB（容量大得多）；单机版至少做"数据量预警 + 自动归档旧年"。

### 6. 无 schema 版本 / 迁移机制
- `STORE_KEY` 写死 `'rikou_v2'`，`load()` 只有 v1→v2 的兜底合并。未来加 `book` / `cashflow` 等字段时，旧数据靠 `Object.assign(base, d)` 兜底，缺字段可能脏。
- 建议：引入 `schemaVersion` 字段 + 显式迁移函数数组（`migrations[from→to]`），每次结构变更可回溯。

### 7. 方案定义了但没落地的功能（设计文档 vs 代码偏差）
对照《账本结构方案.md》：
| 方案决策 | 代码现状 | 状态 |
|---|---|---|
| 决策3：双视角占比条（会计视角 + 穷爸爸现金流视角） | 仅有 `richSplit()` **死代码**（定义后从未被调用），UI 无切换 | ❌ 未落地 |
| 决策5：每笔加 `book` 字段预留多账本 | `ledger` 保存无 `book` 字段 | ❌ 未落地 |
| 结构树 B：交易类型枚举（转账/还款/借出/收回/报销/退款/分期） | 仅有 支出/收入 两类 | ❌ 未落地 |
| 结构树 C：二级分类、图标网格自定义 | 仅一级 `CATS` | ❌ 未落地 |
| 决策4：价格目录归位「账」 | `priceCardHTML` 已在 `moneyStatsHTML` 内 | ✅ 已落地 |
| D 预算：分类预算 | 仅总预算 | ⚠️ 部分 |

> **关键发现**：`richSplit()` 是死代码（grep 确认全项目仅定义处出现）。这是典型"方案写得好、实现走偏"的隐形技术债，也意味着"穷爸爸视角"这个你最看重的创新点**实际没有做**。

### 8. 零自动化测试
- 项目无任何测试套件，全部靠手动点击。当前改存储层（app 化准备）时**极易回归且无法发现**。
- 建议：至少补"数据层单测"（load/save/金额计算/evalExpr/余额重算）+ 关键交互的 DOM 快照测试。本次审查中我临时写的 `_t*.js` 沙箱可改造为正式测试。

### 9. 笔记正文不可搜索
- `notesHTML` 搜索只匹配 `title + sub`，**不搜 `body`**。你之前说"笔记有点乱"，搜索能力弱会加剧这种感受。
- 建议：搜索扩展为正文全文；或建索引。

### 10. 全局记账状态跨表单污染风险
- `ledType/ledCat/ledAcct/ledExpr/ledChannel/ledFocus` 均为模块级全局。`openLedgerForm` 虽会重置，但编辑中途异常跳出、或快速连续打开不同表单时，可能残留旧值导致串台。
- 建议：把记账临时状态收敛进一个 `ledDraft` 对象，随表单生命周期创建/销毁。

### 11. 主题首屏闪白（FOUC）
- `index.html` 写死 `data-theme="light"`，`auto` 暗色用户首屏先白后暗。
- 建议：内联一段 `<script>` 在 `<head>` 里根据 `localStorage` 提前设 theme。

## 三、各维度明细

### PM / 需求
- ✅ 功能广度足够（今天/账/记/设置四层齐全）。
- 🟠 需求缺口：双视角占比、多交易类型、二级分类、分类预算——方案已规划但未实现（见 7）。
- 🟠 "提醒"需求被误读为系统闹钟（见 3）。
- 🟡 首次体验尚可（默认 2 提醒 + 4 账户），但账户余额全 0，新用户难感知价值。

### 运营
- 🔴 数据丢失 = 运营信任崩塌（见 1）。
- 🟠 无留存/回流机制：纯本地，无账号体系，用户流失即数据流失。
- 🟡 导出 JSON 是唯一留存手段，但引导不足。

### 架构
- ✅ 动作查表 `ACTIONS` 设计清晰、可单测、易扩展（加功能=加一行）。
- ✅ 视图/数据/动作三层分离基本到位。
- 🟠 存储层读写虽集中，但 `load()` 内迁移逻辑与 `defaults()` 强耦合，app 化换存储需重写（见 6）。
- 🟠 `save()` 同步 `JSON.stringify` 大对象阻塞主线程（数据量大时卡顿）。

### 前后端
- 项目本就是"无后端"定位，符合设定。
- 🔴 但"app 化跨设备"硬需求下，**零同步能力**是根本局限（之前已与你确认）。
- 🟡 若要后端，建议 Supabase（Postgres + 行级安全 + 实时），改动集中在 `load/save` 两个函数即可。

### 数据库（= 本地存储）
- 🔴 余额快照不可重算（见 2）。
- 🔴 5MB 上限（见 5）。
- 🟠 删除账户后流水悬挂（见 4）。
- 🟠 无 schema 版本（见 6）。
- 🟡 `richSplit` 死代码（见 7）。

### UI / UX
- ✅ 钱迹化记账（内置键盘、两位小数、折叠快选条）体验已顺。
- 🟠 **主页速记账仍唤起系统键盘 + 金额不格式化**：与记账页体验割裂（你之前明确要"不唤起键盘"，但主页 `q-amt` 是 `type=number`）。两处金额交互不一致。
- 🟠 主题首屏闪白（见 11）。
- 🟡 FAB 在窄屏可能遮挡；确认弹窗 `z-index:90` 低于 toast `100`，toast 可能盖住确认框。
- 🟡 内容库排序：media/todo/reminder 的 `date` 多空串，排序不稳定。
- 🟡 无障碍：无 focus trap、aria 有限、无 CSP。

### 测试
- 🟠 零自动化测试（见 8）。
- 🟡 `evalExpr` 边界（`12+` 无右操作数）保存时会 toast 错误而非存错，逻辑 OK 但 UX 提示可优化。
- 🟡 浮点用 `Math.round(amt*100)/100` 修正，基本可靠。

### 安全
- ✅ 所有用户输入经 `esc()` 转义，innerHTML 注入风险低。
- 🟡 无 `Content-Security-Policy`（PWA 建议加）。
- 🟡 `exportObsidian` 的 frontmatter 用 `.replace(/"/g,"'")` 处理 mood，若 mood 含单引号可能破坏 markdown 结构（本地导出，风险低）。
- 🟡 导入 `doImport` 直接 `JSON.parse` + `Object.assign`，无 schema 校验（本地导入自己文件，风险低）。

## 五、对"app 化"的专项判断

你提到"为了后续 app 化做准备"，逐条对应：

| app 化关注点 | 当前状态 | 准备建议 |
|---|---|---|
| 存储跨端 | ❌ 纯 localStorage 单端 | P1 抽 `store.js` + 云同步 |
| 数据可信 | ⚠️ 余额快照不可重算 | P0 加重算 |
| 代码可维护 | ✅ ACTIONS 解耦良好 | 保持，补测试 |
| 离线能力 | ✅ Service Worker 已就绪 | 迁移 IndexedDB 后更强 |
| 原生能力（通知/相机） | ❌ 无 | 用 Capacitor 包裹，仅改 `load/save` 与通知层 |
| 多端账号 | ❌ 无 | Supabase Auth |

**核心结论**：当前架构"业务代码 ↔ 存储"已通过 `load()/save()` 收敛，app 化时**改这两个函数即可换底层**，这是最好的基础。但在此之前，先把 **P0（余额重算 + 数据安全沟通）** 做了，否则 app 化会把"数据不对/会丢"的坑放大到多端。
